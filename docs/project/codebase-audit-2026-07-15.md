### PKCS#11 security review

The pre-fix provider had the following release-blocking defects:

- `C_CreateObject` receives `crypto.randomBytes(32)` as `CKA_VALUE`, so raw key material exists in
  application memory instead of being generated inside the HSM.
- `CKM_AES_CBC_PAD` provides no built-in authentication. The optional “binding” is an HMAC keyed by
  caller-provided context, and ciphertext without context has no integrity protection. This violates
  the repository's mandatory AES-256-GCM rule.
- The template omits explicit AES key type/value-length attributes and treats every algorithm hint,
  including `ECDSA-P256`, as the same secret-key object.
- Tests cover only missing-library/provider-selection behavior; no mock-token or SoftHSM operation
  test protects generation, AEAD round trips, tamper rejection, or non-extractability.

## Upgrade plan (implemented)

### Stability

1. **CI-4:** copied every workspace manifest and the shared-types exported source before the Docker
   frozen install; regression tests derive expected workspaces from the root manifest and require
   the export source copy.
2. **E2E-2:** aligned mock user types with the API onboarding contract, fixed the completion fixture,
   and removed the command-palette race using deterministic readiness assertions.
3. Keep branch protection enabled and treat any red required check as a release blocker.

### Security

1. **CRYPTO-2:** correct handle types; generate non-extractable AES keys with `C_GenerateKey`;
   switch to `CKM_AES_GCM` with 96-bit IV, 128-bit tag, and caller context as AAD; reject malformed
   or tampered payloads; reject unsupported asymmetric hints until a real key-pair implementation
   exists.
2. Add an injected mock PKCS#11 module for deterministic unit coverage and retain a documented
   SoftHSM integration gate for production qualification.
3. Complete operator-only `SEC-ROT` and `MIG-3` before launch.

### Scalability

1. **PERF-3:** made the refresh load scenario assign a distinct seeded account to each VU, then
   acquire and rotate a usable credential without weakening production cookie policy; separated
   hot-read, status, refresh, and CPU-bound Argon2 latency budgets.
2. Stop presenting relaxed GitHub-runner thresholds as the production SLO. Record staging and
   production p95/p99 separately in the maintenance scorecard.
3. Add a sustained soak profile and DB/Redis saturation metrics before raising arrival rates.

### Maintainability

1. **DX-4:** retained the current Biome release, disabled the unstable scanner-activating nursery
   rules, made panic output fail CI, and began incremental test formatting/lint enrollment.
2. **STR-5:** split providers into `src/crypto/hardware-key-store/` with a small selector module and
   kept optional dependency types next to the PKCS#11 adapter.
3. Generate or share `AuthMe` fixture types so API/UI/E2E onboarding fields cannot drift silently.
